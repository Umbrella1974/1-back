现在只做最小修复，不要重构整个系统。

背景：
当前真实 vibration-only 测试中，Python 日志显示 release command_id=2 已经 sent，但受试者不明显感受到 release。同时 slip 的体感持续时间远长于配置的 duration_ms: 1000，像是一直持续到实验结束。

第一性原理判断：
当前 vibration TCP 协议只发送离散整数命令，例如：

* contact: command_id=1
* release: command_id=2
* slip_start: command_id=3

Python 里的 duration_ms 目前只影响 scheduler 何时安排下一个 haptic event，并不一定让 ESP32 在 duration_ms 后停止当前 vibration。如果 ESP32 把 slip_start=3 当作 latched/start command，那么必须显式发送 slip_end=4，否则 slip 会一直持续并盖住 release。

目标：
最小修改，支持 vibration event 的可选 end command，并支持 release 后继续记录手部 pinch 一段时间。

不要修改：

* exp2
* manus_vive_com
* MANUS TCP 8888 接口
* calibration 逻辑
* nback_timeline 核心逻辑
* matrix 发送逻辑
* vendor_exp2_abc worker/protocol

不要重构：

* 不要重写 haptic scheduler
* 不要重新设计 session runner
* 不要新增复杂窗口
* 不要做离线 replay

修改 1：haptic plan 支持可选 end command

在 haptic_plan_config.py 中给 HapticPlanEvent 增加可选字段：

* end_command_label: str | None = None
* end_command_id: int | None = None

YAML 示例：

events:

* name: slip
  modality: vibration
  command_label: slip_start
  command_id: 3
  end_command_label: slip_end
  end_command_id: 4
  duration_ms: 1000
  trigger_zone: closed_zone
  onset_gap_after_previous_ms: [5000, 10000]

校验规则：

1. end_command_id 只对 modality=vibration 的 event 生效。
2. matrix event 如果写 end_command_id，直接报错。
3. end_command_id 必须是 int。
4. end_command_label 可选，仅用于日志。
5. 不要求所有 vibration event 都有 end_command_id；只有配置了才发 end command。
6. to_dict() 要能 round-trip end_command_label 和 end_command_id。

修改 2：ScheduledHapticEvent 携带 end command 信息

在 haptic_trial_scheduler.py 的 ScheduledHapticEvent 中增加：

* event_end_monotonic_ms
* end_command_label
* end_command_id

event_end_monotonic_ms = adjusted_onset_ms + sampled_duration_ms

注意：
不要改变现有 event onset 调度逻辑。
不要因为 end_command_id 而生成新的 haptic plan event。
这只是给 sender 一个“到时间发停止命令”的信息。

修改 3：SimpleHapticSender 支持 pending end command

在 simple_haptic_sender.py 中增加一个很小的 pending end command 机制。

建议新增 dataclass：

PendingVibrationEndCommand:
due_monotonic_ms: float
source_event_name: str
command_label: str | None
command_id: int
haptic_trial_index: int
event_index: int

SimpleHapticSender.record_scheduled_event(scheduled):

1. 按原逻辑发送主 command_id。
2. 如果 scheduled.modality == "vibration" 且 scheduled.end_command_id is not None：

   * 不要立刻发送 end command。
   * 把 PendingVibrationEndCommand 放入 self._pending_end_commands。
   * due_monotonic_ms = scheduled.event_end_monotonic_ms。

新增方法：

poll_due_control_commands(now_ms: float) -> list[HapticEventRecord]

行为：

1. 找出 due_monotonic_ms <= now_ms 的 pending end commands。
2. 对每个 due command，通过 vibration worker 发送 end_command_id。
3. 写日志或至少打印：
   [HAPTIC] end_command source=slip command_id=4 tcp=...
4. 从 pending list 中移除。
5. 返回记录，方便测试。

主循环修改：
在 run_pinch_haptic_1back.py 的正式阶段主循环中，每一轮都调用：

sender.poll_due_control_commands(now_ms)

在程序退出前也调用一次 drain/stop，确保 due 的 end command 已发完。
不要用 threading.Timer。用主循环 poll，方便测试，也不会在程序退出时丢命令。

修改 4：日志字段

haptic_events.csv 增加这些字段：

* event_end_monotonic_ms
* end_command_label
* end_command_id
* end_command_due_ms
* end_command_sent
* end_command_tcp_success
* actual_duration_ms

最小实现可以这样：

* 主事件 slip 行记录 end_command_id=4、end_command_due_ms、end_command_sent=false/true。
* 或者额外追加一行 event_name=slip_end。
  两种都可以，但必须能从 CSV 看出：
  slip_start 什么时间发；
  slip_end 什么时间发；
  两者间隔是否接近 duration_ms。

优先推荐额外追加一行：

* event_name=slip_end
* modality=vibration
* command_id=4
* source_event_name=slip
* send_status=sent/queued/failed

如果这个改动太大，则先在 slip 行里记录 end_command_sent 和 actual_duration_ms。

修改 5：release 后摇记录

在 dualtask_config.yaml 增加：

session:
post_release_recording_ms: 3000
post_release_continue_nback: false

语义：

1. release event emit 后，不要立刻 break 主循环。
2. 设置 post_release_end_ms = release_emit_ms + release_duration_ms + post_release_recording_ms。
3. 从 release emit 到 post_release_end_ms：

   * 继续接收 MANUS raw frame；
   * 继续写 raw_frames.jsonl；
   * 继续写 pinch_timeseries.csv；
   * 不再触发新的 haptic event；
   * 不再开始新的 haptic trial；
   * 如果 post_release_continue_nback=false，不再出现新的 1-back 数字，只显示空白或 “Recording hand data...”。
4. 到 post_release_end_ms 后结束 session。

summary.json 增加：

* post_release_recording_ms
* post_release_continue_nback
* post_release_started_ms
* post_release_end_ms
* post_release_pinch_samples
* end_reason

end_reason 推荐：

* release 刚 emit 时不要立刻写 final end。
* post-release 完成后写：
  end_reason = "haptic_release_post_recording_complete"

测试要求：
新增或修改测试：

1. haptic_plan_config 能解析 end_command_id / end_command_label。
