现在只修一个问题：slip_start=3 发出后，没有在 duration_ms 后发送 slip_end=4。

不要改 MANUS TCP。
不要改 calibration。
不要改 1-back。
不要改 haptic scheduler 的事件顺序。
不要改 matrix。
不要重构 runner。

现象：
真实 vibration 测试中，haptic_events.csv 里：
- slip command_id=3, duration_ms=1000, tcp_success=true, send_status=sent
- release command_id=2, tcp_success=true, send_status=sent
- 但是 slip 行 end_command_id 为空，end_command_sent=False
- 没有任何 slip_end 行
体感上 slip 一直持续到实验结束，release 不明显。

判断：
duration_ms 目前只用于 scheduler 计算下一个事件时间。
它没有让 ESP32 自动停止 slip。
必须在 slip duration 到期后显式发送 slip_end command_id=4。

第一步：确认 plan 文件
在当前实际使用的 haptic_plan_config_example.yaml 里，slip event 必须写：

- name: slip
  modality: vibration
  command_label: slip_start
  command_id: 3
  end_command_label: slip_end
  end_command_id: 4
  duration_ms: 1000
  trigger_zone: closed_zone
  onset_gap_after_previous_ms: [5000, 10000]

如果实际运行用的是别的 plan，也必须在那个 plan 的 slip event 加 end_command_id: 4。
不要只改示例文件，必须确认 dualtask_config.yaml 里的 haptic_plan_config 指向的实际 plan 文件也加了。

第二步：实现真正发送 end command

当前 haptic_plan_config.py 已经有 end_command_id 字段，但 simple_haptic_sender.py 没有真正发送它。

请在 simple_haptic_sender.py 中实现：

1. 新增 pending end command 队列，例如：

@dataclass
class PendingVibrationEndCommand:
    due_monotonic_ms: float
    source_event_name: str
    command_label: str | None
    command_id: int
    haptic_trial_index: int
    source_event_index: int

2. 在 record_scheduled_event(scheduled) 中：
   - 正常发送主事件 command_id。
   - 如果 scheduled.modality == "vibration" 且 scheduled.end_command_id is not None：
       把 PendingVibrationEndCommand 加入队列。
       due_monotonic_ms = scheduled.event_end_monotonic_ms
       不要立刻发送 end command。

3. 新增方法：

poll_due_control_commands(now_ms: float) -> list[HapticEventRecord]

行为：
- 找到 due_monotonic_ms <= now_ms 的 pending end commands。
- 对每个到期 command，使用 vibration TCP worker 发送 command_id。
- 生成一条日志记录，event_name = f"{source_event_name}_end"，例如 slip_end。
- command_label = end_command_label
- command_id = 4
- source_event_name = slip
- tcp_enabled/tcp_queued/tcp_success/send_status 正常记录。
- 从 pending 队列移除。
- 返回这些 control records。

4. 在 run_pinch_haptic_1back.py 的正式主循环里，每一轮都调用：

sender.poll_due_control_commands(now_ms)

必须在 release 后 post-release recording 阶段也继续调用，直到 pending end commands 发完。

5. 程序退出前，至少再调用一次 drain/poll，避免到期的 end command 没发。

第三步：日志字段
haptic_events.csv 必须能看到两行：

slip      command_id=3  send_status=sent
slip_end  command_id=4  send_status=sent

或者如果不想新增行，也至少要在 slip 行里写：
- end_command_id=4
- end_command_sent=True
- end_command_tcp_success=True
- actual_duration_ms 接近 1000

但我更建议新增 slip_end 行，因为最直观。

第四步：测试
新增/修复测试：

1. plan 里的 slip 能解析 end_command_id=4。
2. record_scheduled_event(slip) 后不会立刻发送 slip_end，只加入 pending。
3. poll_due_control_commands(now_ms < due) 不发送 slip_end。
4. poll_due_control_commands(now_ms >= due) 发送 slip_end command_id=4。
5. run_pinch_haptic_1back 主循环会持续调用 poll_due_control_commands。
6. post-release recording 阶段也会调用 poll_due_control_commands。
7. haptic_events.csv 中能看到 slip_end 行，且 slip_end.monotonic_ms - slip.monotonic_ms 约等于 duration_ms。
8. release 后 post_release_recording_ms 仍然生效。
